# Denova Frontend Design Guide

本文档定义 Denova 的前端视觉、布局、组件与交互原则。它用于指导新功能、重构、设计评审和视觉验收，不替代业务架构、数据协议或具体组件源码。

Denova 是面向创作者的 AI 创作工作室。界面应像一套安静、可靠、长期使用不疲劳的创作工具：内容是主角，AI 是协作者，产品框架负责组织、解释和保护创作过程。

本指南中的用语：

- **必须 / 禁止**：新代码和重构必须遵守。
- **应该 / 不应该**：除非有明确的产品理由，否则遵守。
- **可以**：在符合上层原则时自行判断。

## 设计命题

Denova 的视觉签名是 **纯色工作室 + 有质感的阅读画布 + 克制的状态信号**。

- 产品框架使用纯黑或纯白、中性灰层级与细边界，长时间使用时主动退后。
- 创作正文、故事插图和用户内容可以有丰富色彩；产品 chrome 不与内容争夺注意力。
- 写作模式与游戏模式共享同一种产品语言。差异来自任务、内容和信息密度，而不是两套互不兼容的皮肤。
- AI 活动必须可理解、可停止、可追踪。界面只陈述真实发生的状态，不用动画或文案伪装确定性。

当不确定如何设计时：

1. 先让创作内容更清楚，再增加产品装饰。
2. 先用间距、字重、表面和边界建立层级，再使用颜色。
3. 先找相邻功能和现有组件，再创建新的视觉模式。
4. 先保证写作与游戏、中文与英文、浅色与深色都成立，再优化单一截图。

## 设计原则

### 1. 内容优先

正文、剧情、设定、图片和用户决策拥有最高视觉优先级。工具栏、目录、Agent 轨迹和辅助元数据应在需要时清楚，在阅读与创作时安静。

- 中央画布是默认焦点；侧栏和辅助面板不能无理由挤压核心内容。
- 高频动作靠近任务发生处；低频动作进入菜单、设置或渐进披露区域。
- 不用大面积品牌色、营销式渐变或装饰性插画填充工作区。
- 空间不足时先收纳辅助信息，不压缩正文到不可读宽度。

### 2. 安静但不含糊

克制不等于低对比、难发现或只有 hover 才可用。主操作、当前上下文、保存状态、错误和可拖拽边界都必须明确。

- 用位置、尺寸、对齐和字重建立主要层级。
- 用背景层级与 1px 边界组织复杂工作台。
- 颜色只承担状态、焦点、危险或少量创作语义。
- 每个视图最多有一个实心高强调主操作。

### 3. 一套系统，两种模式

写作模式与游戏模式共享导航、组件、token、反馈、快捷键和可访问性规则。共通能力的变更必须同时检查两个模式。

- 模式切换只能由用户显式触发。
- 点击书籍、Skills、Agents、自动化、版本管理、设置等共享一级菜单，禁止隐式切换模式。
- 任意时刻只能有一个一级菜单呈现 active 状态。
- 同一动作在两个模式中保持同名、同图标、同反馈和同危险等级。

### 4. 为连续创作优化

Denova 是长时间、高频使用的生产工具，而不是只展示一次的落地页。

- 避免不必要的弹窗、欢迎页、教学遮罩和确认步骤。
- 保持滚动位置、编辑焦点、草稿和面板尺寸稳定。
- 常见流程在系统已有足够信息时，应在 1–2 次操作内完成。
- 长任务可以持续运行；界面提供阶段、停止与恢复线索，不能用隐藏超时替代状态设计。

### 5. 真实、可检查的 AI

- 运行中使用过程语言；只有拿到真实结果后才使用“已保存”“已验证”“已完成”等结果语言。
- 思考、工具调用、日志和最终内容有清晰层级，不能让内部轨迹冒充创作结果。
- 文件修改、版本恢复和 Agent 建议应显示对象、范围与下一步动作。
- 失败状态说明发生了什么、影响了什么、用户可以怎么恢复。

## 设计 token

### 两层 token 边界

当前前端有两层语义 token，职责不同：

1. `--nova-*` 是 Denova 产品层 token，用于工作台、编辑器、对话、树、状态和创作表面。
2. shadcn token（如 `--background`、`--card`、`--popover`、`--primary`、`--muted`）用于 `web/src/components/ui/` 中的通用原语。

新代码必须遵守：

- TSX 中已有语义 token 时，禁止硬编码 hex、rgb、oklch 或重复创建近似颜色。
- 产品功能优先使用 `--nova-*`；通用原语内部优先使用 shadcn token。
- 禁止在某个业务组件内创建第三套全局色板。
- 复杂且真正隔离的子工作台可以定义局部 token，例如 `.preset-workspace`；局部 token 必须有容器作用域，不能泄漏到全局。
- 新增全局 token 时必须同时定义深色和浅色值，并说明稳定的语义角色。名称应描述用途，不描述偶然的色相。
- 需要现有颜色的透明变体时优先使用 `color-mix()`，不要增加只差透明度的新 hex。
- Tailwind 需要消费新 token 时，在 `@theme inline` 中建立绑定。

### 产品颜色角色

| Token 角色 | 用途 | 不用于 |
| --- | --- | --- |
| `--nova-bg` / `--nova-bg-deep` | 应用画布、顶层导航、最深层 chrome | 卡片、输入框、浮层 |
| `--nova-surface` | 侧栏、面板、主要工具表面 | hover 或 active 状态 |
| `--nova-surface-2` / `--nova-surface-3` | 次级区域、嵌入内容、层级分隔 | 无意义的斑马纹 |
| `--nova-hover` | 指针悬停、可点击行的瞬时反馈 | 持久选中状态 |
| `--nova-active` / `--nova-selection-bg` | 当前项、按下态、持久选中 | 大面积页面背景 |
| `--nova-border` / `--nova-border-soft` | 分隔线、控件轮廓、面板边界 | 强调焦点或错误 |
| `--nova-text` | 正文、标题、主要标签 | 禁用与辅助信息 |
| `--nova-text-muted` | 次要标签、描述、图标 | 长篇正文 |
| `--nova-text-faint` | 元数据、占位符、禁用提示 | 关键操作或错误 |
| `--nova-accent` | 中性强调、主要焦点附近的少量信号 | 装饰性铺色 |
| `--nova-success*` | 已完成、有效、新增、可用 | 普通选中状态 |
| `--nova-warning*` | 待处理、冲突、未保存、需要注意 | 一般说明文字 |
| `--nova-danger*` | 错误、删除、不可逆操作，以及需要立即发现的运行停止入口 | 取消、关闭、返回 |
| `--nova-editor-*` | 编辑器画布、文字与编辑器内强调 | 应用 chrome |
| `--nova-dialogue-highlight` | 正文和消息中的对话高亮 | 通用品牌色 |

`--nova-accent-blue` 是现有历史命名，实际承担重点、审阅与分支等强调角色，不能根据名字假设它一定是蓝色。新 token 应按语义命名；涉及该 token 的系统性重命名应一次完成定义和全部调用点，避免长期保留别名。

### 色彩纪律

- 深色主题的主画布以纯黑为基底；浅色主题以纯白为基底。
- 中性色承担绝大多数界面。绿色、黄色、红色采用低饱和、高可读的状态色。
- 故事插图、封面和用户内容不受 chrome 色板限制，但周围容器保持中性。
- 同一状态在写作和游戏模式中使用相同颜色语义。
- 状态不能只靠颜色表达；同时提供图标、文字、形状或位置变化。
- 正文文字与背景至少满足 WCAG AA；小号辅助文字也不能通过过度降低不透明度来“变安静”。
- 渐变只可用于内容、遮罩或明确的空间过渡。禁止把彩色渐变、霓虹光晕和玻璃拟态当作默认 AI 视觉语言。

## 字体与排版

排版承担 Denova 的大部分个性：工具层精确、克制，阅读层舒展、有文学质感。

| 角色 | 当前 token / 实现 | 用途 |
| --- | --- | --- |
| UI 正文 | `--nova-ui-font-family`、14/20 | 按钮、导航、表单、面板正文 |
| UI 小号 | `--nova-ui-compact-*`、13/18 | 路径、计数、次级元数据 |
| UI 微型 | `--nova-ui-micro-*`、12/16 | 紧凑标签、状态补充；只在空间确实受限时使用 |
| UI 强调 | 16/22 | 对话、重要说明、移动端可读文本 |
| 创作正文 | `--nova-reading-font-family`、默认 18px / 1.9 | TipTap 正文与沉浸式阅读 |
| 等宽 | `font-mono` / 系统等宽栈 | 路径、代码、模型 ID、结构化值、对齐数字 |

规则：

- UI 字体优先适配中英文系统字形；创作正文使用现有中文衬线栈，强化“写作画布”而不是“管理后台”的感受。
- 编辑器正文默认最大行宽为 760px。调整时优先保证舒适行长，不为填满屏幕而拉宽文字。
- 多行正文使用更宽松行高；单行标签和控件使用紧凑行高。
- 一个视图通常只使用常规、medium/semibold 和必要的标题粗体，不堆叠过多字重。
- 大标题收紧字距，小号中文和正文不要人为增加字距。
- 需要比较数字时使用 tabular figures；路径、代码和 ID 才使用等宽字体，普通文案不用等宽制造“技术感”。
- `data-nova-app-shell` 会重新映射部分 Tailwind 字号。不要仅凭 `text-xs`、`text-sm` 的 Tailwind 默认值判断视觉尺寸，应在浏览器中确认实际计算值。
- 禁止为了塞入空间而使用 10px 以下文字。空间不足时应改布局、换行、截断并提供可访问的完整信息。

## 间距、形状与深度

### 间距

使用 4px 基础节奏：4、8、12、16、24、32、40、64。

- 4px：图标与极短标签、紧密元数据。
- 8px：同一控件或同一信息组内部。
- 12–16px：控件与控件、卡片内边距、常规工具栏。
- 24px：不同信息组或表单小节。
- 32–40px：独立页面区域。
- 64px：少数需要沉浸留白的空状态或首页区域。

密集工作台可以紧凑，但必须依靠分组和对齐提高扫描速度，不能靠挤压所有间距提高“专业感”。文字、图标、数字、快捷键和尾部操作应落在稳定的列线上。

### 圆角

- `--nova-radius` 是产品控件的基础圆角。
- 使用现有 shadcn 原语时沿用原语圆角，不在调用点逐个覆盖。
- 紧凑控件和 chip 使用小圆角；卡片、菜单与对话框使用中等圆角；全屏表面可略大。
- `rounded-full` 只用于头像、圆形按钮、状态点和真正的 pill。
- 同一视图保持一个圆角家族。禁止同时混用大量直角、超大圆角和胶囊形卡片。

### 边界与阴影

层级优先顺序：**表面色差 → 1px 边界 → 轻微阴影 → 浮层阴影**。

- 大多数工作台区域停留在前两级。
- 卡片通常只需要表面与边界，不需要悬浮阴影。
- popover、menu、tooltip、dialog 和脱离画布的预览才使用 `--nova-shadow` 或原语内置阴影。
- 禁止用多层厚阴影、发光描边或玻璃模糊区分普通区域。
- 避免卡片嵌套卡片。若内容不是可重复对象、独立工具或需要边界保护的单元，优先使用无框分组和分隔线。

## 布局系统

### 桌面工作台

桌面布局围绕以下稳定结构展开：一级活动栏、可选项目侧栏、中央创作画布、可选右侧 Agent / 状态面板，以及少量顶部和底部状态区域。

- 复用 `WorkspaceLayout` 和 `react-resizable-panels`，不要在功能内再实现一套拖拽面板。
- 中央画布获得剩余空间；侧栏以内容可用的最小宽度为下限，并允许用户调整。
- 调整后的面板尺寸应持久化，切换视图后尽量恢复，不产生无理由跳动。
- 宽面板或审阅模式通过已有 emphasis 能力表达，不通过写死 viewport 宽度覆盖整个工作台。
- resize handle 必须有足够命中区域、明确光标、键盘可访问名称和可见 focus 状态。

### 窄屏与移动端

`useIsMobile` 当前在 1023px 及以下切换到紧凑工作台。这个断点反映的是桌面多面板最小可用宽度，不是某一设备型号。

- 复用 `WorkspaceMobileLayout`、`AdaptiveSurface`、`MobilePaneHost` 和 `Sheet`。
- 窄屏把次级面板变成 drawer/sheet，禁止把三栏桌面布局等比缩小。
- 嵌入式复杂工作台优先使用 container query 或 `AdaptiveSurface.collapseAt`，不要只依赖 viewport breakpoint。
- 移动端主要触控目标至少 44×44px；底部主导航使用 48px 级命中区域。
- 处理 `100dvh`、安全区、软键盘、触摸滚动和 overscroll。
- 触屏没有 hover。桌面上 hover 才显示的必要操作，在 coarse pointer 下必须常驻或有等价菜单入口。

### 自适应而非固定尺寸

- 优先使用 `minmax(0, 1fr)`、弹性布局、内容约束和可拖拽面板。
- 固定宽高只用于图标、控制行、可预测的工具栏或有明确阅读宽度的内容。
- 文案区域默认允许扩展或换行；需要截断时提供 tooltip、title 或可展开入口，且完整内容仍可被键盘和触屏访问。
- 同时验证窄屏、宽屏、长文本、空数据和多数据状态。

## 导航与信息架构

- 一级导航的 active 表示“用户当前看到的顶层工作区”，不是最近点击过的模式。
- 写作/游戏模式切换器是唯一可以改变模式的一级入口。
- 共享菜单打开时保留进入前的模式上下文，关闭后返回该模式。
- 同一层级中的菜单项保持一致的高度、图标尺寸、hover、active、focus 和收起行为。
- 高频且互斥的少量视图可用 Tabs；跨任务或包含动作的入口使用导航或菜单，不用 Tabs 伪装。
- 路径和层级较深时使用 breadcrumb；breadcrumb 描述真实层级，不作为装饰编号。
- 选中、键盘高亮和当前项是不同状态。必要时分别使用 `aria-selected`、`data-selected` 和 `aria-current`，不要只靠同一种背景色。

## 组件选择

### 原语优先

优先顺序：

1. `web/src/components/ui/` 中已有 shadcn / Radix 原语。
2. `web/src/components/common/` 或相邻功能中的共享封装。
3. 成熟依赖提供的完整交互方案。
4. 只有现有抽象确实无法表达时，才创建新的可复用原语。

禁止手写已经由 Radix、shadcn、TipTap、Monaco、cmdk、Sonner、react-resizable-panels 等依赖解决的焦点管理、键盘行为或复杂编辑能力。

### 选择正确的浮层

| 需求 | 使用 | 不使用 |
| --- | --- | --- |
| 为图标按钮命名 | `Tooltip` / `TooltipIconButton` | `title` 作为唯一提示、HoverCard |
| 预览包含丰富但非关键的信息 | `HoverCard` | Tooltip 塞入长文或操作 |
| 点击后显示动作列表 | `DropdownMenu` | 手写 Popover 菜单 |
| 右键上下文动作 | `ContextMenu` | 普通 DropdownMenu 冒充右键菜单 |
| 任意轻量内容、选择器、小表单 | `Popover` | 会无必要阻塞任务的 Dialog |
| 必须完成或明确放弃的决策 | `Dialog` | Popover、页面内伪遮罩 |
| 不可逆确认 | `AlertDialog` | 普通 Dialog 中模糊的“确定” |
| 窄屏侧面板或边缘工作区 | `Sheet` | 超宽居中 Dialog |
| 单选已知列表 | `Select` | 自制 listbox |
| 长列表搜索选择 | `Command` + `Popover` | 无搜索的超长 Select |
| 瞬时确认 | Sonner toast | 持久 banner 或 Dialog |
| 用户必须处理的错误 | 就近 inline error | 会消失的 toast |

如果需要大量 CSS 才能让一个原语表现得像另一个原语，应重新选择组件；它们的焦点、关闭和键盘语义不同。

### 按钮与动作层级

- `default`：当前视图唯一的肯定性主操作。
- `secondary` / `outline`：与主操作并列的次级动作、工具栏动作。
- `ghost`：图标按钮、行内动作、低干扰 chrome。
- `link`：正文中的导航或轻量文本动作。
- `destructive`：删除、覆盖、永久丢弃等真实破坏性动作。

规则：

- Cancel、Close、Dismiss 和 Back 是退出路径，不是 destructive。
- Stop 中止当前运行但不删除数据时，不需要破坏性确认；进行中的生成任务可以使用克制的 danger 信号保证停止入口可发现，但不能与“删除数据”混为同一语义。
- 按钮文案描述结果，例如“保存设置 / Save settings”，不用“确定 / OK”。
- 同一工具栏中的按钮尺寸与行高匹配。桌面可使用紧凑尺寸，移动端扩大命中区域。
- 异步提交后立即禁用重复触发，但保持按钮宽度和布局稳定。

### 图标

- 统一使用 `lucide-react`，禁止再引入第二套通用图标库。
- 默认 UI 图标为 16px；元数据可用 12–14px；空状态或重点入口才使用更大图标。
- 保持 Lucide 默认 stroke，不在每个调用点随机调整粗细。
- 图标颜色继承文字语义，不直接硬编码颜色。
- 图标按钮必须有本地化的 accessible name；优先使用 `TooltipIconButton`。
- 不能仅靠陌生图标表达关键动作。空间允许时显示文字，空间受限时提供 tooltip。
- Emoji 不作为导航、状态或功能图标。

### 列表、树与卡片

- Idle 行通常透明；hover 使用 `--nova-hover`；当前项使用 `--nova-active` 并带语义属性。
- 行尾操作在键盘 focus-within 时必须出现；触屏上不能依赖 hover。
- 文件树、分支树和状态树复用既有树行为，保持展开、选择、拖拽与上下文菜单一致。
- Card 只包裹可重复对象、独立工具或需要保护边界的内容。普通页面小节用标题、间距和 separator 即可。
- 数量、快捷键和尾部元数据右对齐；主要文本左对齐。只有图标、状态点和真正对称的单元格居中。

### 表单

每个字段按“标签 → 必要说明 → 控件 → 错误或尾部信息”组织。

- Placeholder 不能代替 label。
- 使用 `aria-invalid`、`aria-describedby` 和原语内置 focus/error 样式。
- 校验错误紧邻字段，说明错误和修复方式；不要只把边框染红。
- 开关用于立即生效的布尔状态；需要提交或包含多个选项时使用表单控件，不用开关伪装按钮。
- 自动保存必须显示“保存中 / 已保存 / 保存失败”的真实状态，并避免每次输入导致页面跳动。
- 高级配置渐进披露，但不能隐藏会影响数据安全或运行结果的关键选项。

## 写作与游戏场景

### 写作模式

- 正文编辑器是页面视觉中心，默认使用阅读字体、舒适行高和有限行宽。
- 目录、版本、审阅批注和 Agent 面板服务于正文，不能持续制造高对比噪音。
- 编辑器 toolbar 保持单行、稳定和可预测；低频排版设置进入设置面板。
- 对话高亮等创作语义可以使用少量暖色，但不能降低正文对比或改变原文含义。
- Agent 建议、批注和 diff 必须与正式正文有明确边界，接受前不能伪装成已写入内容。

### 游戏模式

- 剧情文本、角色选择和故事图片可以形成更沉浸的舞台；周围导航和控制仍沿用同一中性系统。
- 图片色彩属于故事内容，不应反向驱动整套 chrome 变色。
- 选择、回合、分支、记忆和状态变化必须可回看，不能只用短暂动画表达。
- 状态面板优先展示对当前决策有用的信息；导演过程、规则审计和内部数据进入分层详情。
- 游戏模式不使用仿羊皮纸、霓虹 HUD 或过度装饰边框牺牲可读性，除非未来存在明确、可配置且完整实现的主题系统。

### 共通能力

涉及布局、导航、会话、设置、版本、Agent、Skills、资料库和自动化的变更，必须同时检查写作和游戏模式。若两种模式确实需要不同实现，应共享底层原语和状态语义，并在模块边界上明确差异原因。

## AI 状态与长任务反馈

反馈取决于用户感知时长，不取决于后端是否“异步”：

| 感知时长 | 推荐反馈 |
| --- | --- |
| 瞬时 | 不显示 spinner，直接呈现结果 |
| 短暂等待 | 立即防止重复提交，必要时延迟出现轻量 loading |
| 数秒 | Spinner 或动作标签变为具体进行态，例如“正在读取文件…” |
| 长任务 / 多阶段 | 阶段标签、进度或已完成步骤、停止入口、可展开详情 |

这些阈值只控制反馈，不得用于中断模型或工具运行。LLM 任务默认不限运行时间；安全或循环保护需要独立、可配置的产品策略。

- Loading 出现前预留空间，避免按钮、工具栏和消息在运行中改变宽度。
- 使用具体阶段，不用一个无标签 spinner 覆盖整个长任务。
- 流式内容保持滚动稳定；只有用户仍在底部时才自动跟随。
- 用户滚离底部后显示“回到底部”，禁止抢回滚动位置。
- Stop 后明确区分“已停止”“停止请求中”和“已完成”。
- 可重试错误保留用户输入和上下文；重试按钮靠近错误。
- Tool card 默认展示对下一步有用的摘要，调试日志和大对象进入按需展开区域。
- 成功、失败、部分成功和被取消是不同状态，必须穷尽处理，不能用默认分支吞掉新状态。

## 动效

动效用于解释连续性、层级变化和空间关系，不用于证明产品“智能”。

- 使用 `web/src/features/motion/motion-tokens.ts` 中的 `novaEase`、`novaSpring` 和共享 variants。
- CSS 使用 `--nova-motion-fast`、`--nova-motion-normal`、`--nova-motion-slow`，不要在组件中散落相近 duration。
- hover、focus 和简单状态通常无需位移动画；颜色变化足够时不增加缩放。
- 展开、折叠、sheet、popover 和面板切换可以使用短动效帮助理解来源与去向。
- 禁止让正文、输入框、选中行或主要布局在无用户意图时漂移。
- 循环动画只用于真实运行中的进度；环境装饰和永久呼吸光效不属于工作台。
- 必须尊重 `system`、`full`、`reduced`、`off` 四种 Motion 偏好。新增 Motion 动画要在 reduced/off 下验证。

## 文案与国际化

文字是界面的一部分，不是开发完成后的填充物。

- 所有用户可见文本和 accessible name 必须通过 i18next 提供中文与英文，禁止在 TSX 中硬编码任一语言。
- 同一 key 在 `zh-CN` 与 `en-US` 中同时添加；运行 i18n key 检查。
- 英文 UI 默认使用 sentence case；中文使用自然、简洁的现代书面语。
- 动作使用具体动词和对象：“创建书籍 / Create book”“保存设置 / Save settings”，不用“提交”“确定”“处理”。
- 同一动作贯穿触发、进行、完成和错误状态使用同一词根。
- 进行态使用真实动作：“正在生成… / Generating…”；不用泛化的“处理中”。
- Toast 直接说改变了什么，不使用“成功地 / successfully”。
- 错误文案采用“发生了什么 + 影响 + 下一步”，不道歉、不责怪用户，也不只显示错误码。
- 空状态说明为什么为空，并给出第一个可执行动作；无可用动作时再解释限制。
- 文案不能过度宣称模型已理解、已保护、已验证或已保存，除非有对应状态数据。
- 使用本地化的日期、时间、数字、复数和标点。不要通过字符串拼接制造中英文语序问题。
- 检查英文较长标签、中文无空格长文本、文件名、模型名和用户输入造成的换行与截断。

## 可访问性

- 所有交互使用正确的 `button`、`a`、input、menu、dialog 等语义，不用带 click 的普通 `div` 代替。
- 每个交互元素必须有可见 `:focus-visible`。禁止移除 outline 后不给等价 focus ring。
- 核心流程可以只用键盘完成；Dialog/Popover 打开后聚焦最可能使用的控件，Enter 与 Esc 行为符合预期。
- Icon-only 按钮有本地化 `aria-label` 和 tooltip；状态图标有文本等价物。
- 当前项、展开态、选中态、忙碌态和错误态使用对应 ARIA 属性。
- 文字与背景满足 WCAG AA；状态不只依赖红绿差异。
- 图片根据用途提供 alt：内容图片描述与任务有关的内容；纯装饰图片使用空 alt。
- 触控目标、滚动、缩放、软键盘和安全区必须可用。
- 尊重 reduced motion；自动滚动、闪烁和持续动画不能妨碍阅读。
- 屏幕阅读器文案也属于双语范围，不能把英文 `Close`、`Loading` 等遗留在原语中。

## 状态设计

每个新视图在实现前至少考虑：

| 状态 | 需要回答的问题 |
| --- | --- |
| 初始 / 空 | 用户为什么在这里？第一步是什么？ |
| Loading | 正在加载什么？是否需要占位，布局会不会跳？ |
| Streaming / Running | 当前阶段是什么？能否停止？已有结果是否可用？ |
| 成功 | 什么对象发生了变化？界面是否已经反映结果？ |
| 部分成功 | 哪些完成、哪些失败？如何继续？ |
| 错误 | 发生了什么？输入和上下文是否保留？如何重试或修复？ |
| Disabled | 为什么不可用？用户如何满足条件？ |
| Empty search | 是没有数据，还是筛选无结果？如何清除筛选？ |
| Long content | 是否换行、滚动、截断或按需展开？ |
| Offline / reconnecting | 当前内容是否安全？恢复后会发生什么？ |

Skeleton 只用于结构已知且即将填充的内容。未知长任务不使用无限 skeleton；显示真实阶段和稳定容器。

## 配置边界

较大的体验变化要判断是否需要配置，但不是每个视觉选择都暴露给用户。

适合配置的例子：主题、语言、Motion 强度、编辑器字号与行高、与创作流程直接相关的面板偏好。

不适合配置的例子：圆角大小、普通按钮颜色、组件间距、阴影强度、同一语义的多套状态色。这些属于设计系统，应保持一致。

新增配置时必须明确默认值、用户级或工作区级作用域、持久化位置，以及写作和游戏模式是否共享。

## 禁止的模式

- 通用 AI 渐变、霓虹发光、玻璃拟态和无语义的彩色光斑。
- 为每个小节加 Card，或在 Card 中继续嵌套 Card。
- 在 TSX 中硬编码颜色、用户文案、z-index 体系或动画时长。
- 为单个页面复制 Button、Dialog、Select、Tooltip、Toast 或文件树。
- 依赖 hover 暴露唯一入口，或用 tooltip 承载错误和关键说明。
- 用 destructive 样式表达取消、返回、关闭和普通停止。
- 在共享一级菜单点击时自动切换写作/游戏模式。
- 同时点亮多个一级菜单，或只用颜色表示当前项。
- 把桌面三栏强行缩到手机，或用固定像素宽度假装 responsive。
- 长任务只显示一个无限 spinner，没有阶段、停止、错误或恢复信息。
- 让流式消息、图片加载或状态更新抢走焦点和滚动位置。
- 用“已完成”等结果文案描述尚未确认的模型意图。
- 为了视觉紧凑牺牲 12px 以下文字、键盘 focus、触控面积或中英文完整性。

## 实现流程

### 开始前

1. 写清用户当前任务、主要内容和唯一主操作。
2. 标记功能属于写作、游戏还是共通能力；共通能力默认覆盖两种模式。
3. 找到最接近的相邻视图、共享原语和现有 token。
4. 画出空、加载、运行、成功、错误和窄屏状态，而不只画理想数据截图。
5. 判断体验变化是否需要配置，以及配置作用域。

### 实现中

1. 复用组件和布局抽象，保持模块职责清晰。
2. 使用语义 token，同时完成 light/dark。
3. 同时添加 zh-CN/en-US 文案与 accessible name。
4. 穷尽异步状态，保护草稿、焦点、滚动和用户内容。
5. 为交互行为补测试；不要为静态样式常量写低价值测试。

### 浏览器验收

前端可见变更必须打开真实页面验证。已有热加载服务时直接使用，不另起进程抢占端口。

最低检查矩阵：

- 写作模式与游戏模式；共通页面从两种模式分别进入和返回。
- 深色与浅色。
- 中文与英文。
- 约 390px 窄屏、768–1023px 紧凑宽度、1440px 及以上桌面宽度。
- 空数据、典型数据、长文本、大量数据、loading、streaming、错误、disabled。
- 键盘 focus、Enter/Esc、触屏无 hover、reduced/off Motion。
- 面板拖拽、软键盘、滚动锁、长列表与流式追加。

建议的前端检查：

```bash
cd web
pnpm check:i18n
pnpm test
pnpm build
```

仓库级交付仍以项目构建流程为准。

## 设计评审清单

评审前端页面时按影响从高到低回答：

1. **任务**：用户能否立即看懂当前位置、主要内容和下一步？
2. **层级**：主操作是否唯一？辅助 chrome 是否抢过内容？
3. **效率**：高频流程是否有无意义的中间步骤、弹窗或重复输入？
4. **一致性**：是否复用相邻组件，并同时适用于写作与游戏模式？
5. **状态**：空、长任务、部分成功、失败和恢复是否完整、真实？
6. **自适应**：窄屏是重新编排还是简单压缩？长文本与大数据是否稳定？
7. **可访问性**：键盘、focus、触屏、对比度和 reduced motion 是否成立？
8. **国际化**：中英文是否完整、自然，英文变长后是否仍可用？
9. **维护性**：是否使用语义 token、现有原语和清晰模块边界，而非局部补丁？

评审意见应指出具体元素、具体摩擦和可执行修改，优先列出最影响创作流程的 3 项，不用抽象的“更高级”“更有设计感”代替判断。
